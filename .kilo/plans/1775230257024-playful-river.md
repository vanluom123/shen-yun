# Implementation Plan: Push Notifications for New Registrations (Admin)

## Goal
Implement push notifications for admin when new user registrations occur, using OneSignal for web/mobile delivery. Notifications should be real-time, appear as system notifications, and not require any UI elements like panels or bells in the admin interface.

## Requirements
- Admin receives push notifications on mobile/web when accessing the admin area.
- Notifications include basic registration details (name, total guests).
- Use existing OneSignal configuration in .env.
- Compatible with deployed web application.
- No additional UI components needed.

## Current State Analysis
- Laravel app with Vite frontend.
- OneSignal keys configured in .env: ONESIGNAL_APP_ID, ONESIGNAL_REST_API_KEY.
- Admin authentication via session (no User model).
- Registration flow in RegistrationWizardController::submit().

## Implementation Steps

### 1. Frontend Integration (OneSignal SDK)
- Add OneSignal SDK script to admin layout (resources/views/layouts/app.blade.php).
- Initialize OneSignal with app ID and prompt for push permission on admin pages.
- Ensure HTTPS in production for push notifications.

### 2. Backend Notification Sending
- Install OneSignal package (optional) or use HTTP client.
- Modify RegistrationWizardController::submit() to send push notification via OneSignal API after registration creation.
- Configure OneSignal service in config/services.php.

### 3. Configuration Updates
- Add OneSignal config to config/services.php.
- Ensure mailer is configured for fallback (existing).

### 4. Testing and Deployment
- Test on localhost (allow localhost as secure origin).
- Deploy and test on production domain.
- Verify push permissions and delivery.

## Dependencies
- OneSignal account and app setup (already done).
- HTTPS required for production push notifications.
- Admin must grant push permission in browser/mobile.

## Risks and Mitigations
- Push blocked by user: Fallback to email (already implemented).
- OneSignal API limits: Monitor usage.
- Browser compatibility: Test on target devices.

## Estimated Effort
- Frontend: 1 hour
- Backend: 2 hours
- Testing: 1 hour
- Total: 4 hours

## Success Criteria
- Admin receives push notification immediately after new registration.
- Notification displays on mobile/web without app UI changes.
- No errors in logs; graceful fallback if push fails.</content>
